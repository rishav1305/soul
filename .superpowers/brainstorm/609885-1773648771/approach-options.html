Mobile Chat — Approach

Two strategies for fixing the mobile layout. Which direction?

A

Surgical — Tailwind Responsive Only

Add sm: and md: variants to existing components. No new components, no layout restructuring. Fix hard-coded widths, add safe-area padding, hide labels on mobile, wrap toolbar to 2 rows.

Pros

  • Minimal code changes (~6 files)
  • No risk of breaking desktop
  • Fast to implement (1-2 hours)

Cons

  • Toolbar still feels cramped, just less broken
  • No mobile-specific UX improvements
  • Desktop and mobile share same component logic
B

Mobile-First Redesign (Recommended)

Rethink the input area for touch. Two-row toolbar: primary actions top, secondary in overflow. Collapsible top bar. Tighter bubbles. Safe areas everywhere. Still the same components — just smarter responsive behavior.

Pros

  • Actually pleasant to use on phone
  • Proper touch targets (44px minimum)
  • Chat input feels native-app quality
  • Scales cleanly across 320px → 1440px

Cons

  • More changes (~8-10 files)
  • Needs careful testing at each breakpoint
  • ~3-4 hours implementation

What Approach B Includes

1. ChatInput 2-row toolbar — Row 1: textarea + send. Row 2: icon-only buttons (product, mode, think, attach) with overflow menu for less-used items
2. Top bar collapse — On mobile: just hamburger + session title + one action button. "New/Running/History" move into sidebar
3. Responsive widths — Replace all hard-coded px with max-w-full sm:max-w-[380px] pattern
4. Safe area padding — Apply env(safe-area-inset-bottom) to ChatInput, sidebar, and bottom-fixed elements
5. Tighter message bubbles — max-w-[92%] sm:max-w-[85%], reduce padding on mobile, scale metadata text up slightly
6. Touch target enforcement — All interactive elements ≥44px on mobile, larger tap areas on send/attach
